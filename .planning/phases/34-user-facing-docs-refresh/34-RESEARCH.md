### GIF capture approach

All three demos can run headlessly (`--headless`) and use `SurgicalEnv.render(mode="rgb_array")`
internally for offscreen rendering. The recommended capture strategy is a lightweight standalone
script `demos/capture_demo_gif.py` (or an extension to `demos/_common.py`) that:

1. Imports the same `_omp_compat` / `_platform_guard` shims as the demos.
2. Loads the demo scene via `demos._common.resolve_scene()`.
3. Constructs a `SurgicalEnv` with MuJoCo backend (reliable headless rgb_array).
4. Runs a scripted policy or the demo's training agent for ~30s at the simulator frame rate.
5. Collects `rgb_array` frames and writes them with `imageio.mimsave()` (or PIL) to
   `docs/demos/{task}.gif`.

**Dependencies:** `imageio` is already a transitive dependency via `phi-flow`/`scikit-image`;
if missing, add it under the `[gui]` or `[simulation]` extra. If the demos are too slow for
real-time 30s capture, reduce physics substeps or run at a lower FPS (e.g., 10 FPS) and
use a shorter episode; the requirement is "~30s" of visual content, not wall-clock duration.

### Risk notes

- **GIF capture variance:** MuJoCo offscreen rendering on macOS stock Python can differ from
  Linux/CI; GIFs are best-effort visual artifacts, not deterministic regression targets.
- **Qt screenshot portability:** Screenshots from Phase 33 are PNGs produced by `QWidget.grab()`
  under the offscreen platform; they are adequate for README but may not show live viewport
  animation.
- **File size:** Three ~30s GIFs can be large; recommend 320-480px width and a 128-color
  palette to keep each GIF under ~5 MB.

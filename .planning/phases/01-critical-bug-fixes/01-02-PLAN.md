**BUG-04 — RewardConfig / create_default_reward**
- `RewardConfig` is a plain `@dataclass` (not Pydantic), so `@field_validator` does not apply. Defaults are negative (`collision_penalty: float = -10.0`).
- `create_default_reward` wraps penalties with `abs()`: `CollisionPenalty(weight=abs(config.collision_penalty), ...)`.
- This masks a sign bug: if a user sets `collision_penalty` to a positive value, `abs()` still makes it positive, violating the contract that penalties must produce non-positive rewards.
- Fix options:
  A. Convert `RewardConfig` to a Pydantic `BaseModel` with validators.
  B. Add `__post_init__` validation in the dataclass.
  The preferred approach is **A** because the rest of the project uses Pydantic v2 and `test_rewards.py` already imports `RewardConfig` as if it were a normal class.
- After fixing validation, remove `abs()` in `create_default_reward`.
- Update `test_rewards.py::TestCreateDefaultReward::test_collision_penalty_is_negative` to instantiate with defaults and assert the composite still produces non-positive penalties.
- Add a regression test asserting that instantiating with a negative penalty raises an error.

**BUG-06 — CurriculumScheduler.apply_parameters (partial application)**
- `apply_parameters` already applies gravity, mass ratio, friction, damping, and stiffness.
- It does **NOT** apply sampled `dynamics` parameters: `action_noise`, `joint_noise`, or `delay` (from `snapshot.dynamics`).
- These values are sampled in `sample_parameters()` but silently ignored in `apply_parameters()`.
- Fix: add dynamics-application block in `apply_parameters`. Since the simulator may not have direct setters for these, log them at `logger.debug` minimum, and apply to the simulator if it exposes `set_body_property` with a "dynamics" namespace or has a matching attribute. If no API exists, at least store them on the scheduler so downstream code can access them.
- Add a stronger regression test in `test_dynamics.py` that passes a snapshot with `action_noise` and asserts it was consumed/not silently dropped.

**BUG-05 — Vision prompts** (excluded from this plan)
- `get_image_to_scene_prompt` already uses `json.dumps(schema_example, indent=2)` which produces valid double-quoted JSON.
- `test_vision_prompt_contains_valid_json` already passes.
- No code change required.

**BUG-07 — LightConfig validator mutates input dict**
- `validate_light_type(mode="before")` does `data["direction"] = (0.0, 0.0, -1.0)` without copying the dict.
- In Pydantic v2, raw input dicts should not be mutated in place.
- Fix: add `data = dict(data)` at the top of the validator so all subsequent mutations are on a copy.
- Add a regression test in `test_schema.py` that passes a dict, constructs `LightConfig(**data)`, and asserts the original dict is unchanged.
</context>

GHCR push requires:
- `docker/login-action` with `registry: ghcr.io`, `username: ${{ github.actor }}`, `password: ${{ secrets.GITHUB_TOKEN }}`
- `docker/metadata-action` to generate tags/labels from git refs
- `docker/build-push-action` with `platforms: linux/amd64,linux/arm64` and `push: true`

Cross-arch build in CI uses QEMU:
- `docker/setup-qemu-action@v3` to emulate arm64 on amd64 runner
- `docker/setup-buildx-action@v3` to create buildx builder
- Build Dockerfiles with `push: false` (verify only, no registry push)

NOTE: ROCm Dockerfile and CUDA Dockerfile are locked to linux/amd64 (see plan 11-01). They build in CI as amd64-only verification. Jetson Dockerfile is arm64-only; it builds only in arm64 verification steps.

Add the following job block as the last job in the file (after the `test` job's closing `steps:` block, at the same indentation level):

```yaml
  docker-ci:
    name: Docker Build (multi-arch)
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up QEMU
        uses: docker/setup-qemu-action@v3

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Build CPU (amd64 + arm64)
        uses: docker/build-push-action@v6
        with:
          context: .
          file: ./Dockerfile
          platforms: linux/amd64,linux/arm64
          push: false
          cache-from: type=gha
          cache-to: type=gha,mode=max

      - name: Build CUDA (amd64)
        uses: docker/build-push-action@v6
        with:
          context: .
          file: ./Dockerfile.cuda
          platforms: linux/amd64
          push: false
          cache-from: type=gha
          cache-to: type=gha,mode=max

      - name: Build ROCm (amd64)
        uses: docker/build-push-action@v6
        with:
          context: .
          file: ./Dockerfile.rocm
          platforms: linux/amd64
          push: false
          cache-from: type=gha
          cache-to: type=gha,mode=max

      - name: Build Jetson (arm64)
        uses: docker/build-push-action@v6

          context: .
          file: ./Dockerfile.jetson

          cache-to: type=gha,mode=max
```

Extend `.github/workflows/release.yml` by adding a new `docker-release` job AFTER the existing `build-and-publish` job. This job builds and pushes multi-arch Docker images to GHCR on `v*` git tag pushes.

Add the following job block as a new job in the file (same YAML indentation level as `build-and-publish`, after that job's closing `steps:`):

```yaml
  docker-release:
    name: Push to GHCR
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up QEMU
        uses: docker/setup-qemu-action@v3

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Docker meta (CPU)
        id: meta-cpu
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/${{ github.repository }}
          tags: |
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=sha,prefix=

      - name: Build and push CPU (multi-arch)
        uses: docker/build-push-action@v6
        with:
          context: .
          file: ./Dockerfile
          platforms: linux/amd64,linux/arm64
          push: true
          tags: ${{ steps.meta-cpu.outputs.tags }}
          labels: ${{ steps.meta-cpu.outputs.labels }}

      - name: Docker meta (CUDA)
        id: meta-cuda
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/${{ github.repository }}/cuda
          tags: |
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}

      - name: Build and push CUDA (amd64)
        uses: docker/build-push-action@v6
        with:
          context: .
          file: ./Dockerfile.cuda
          platforms: linux/amd64
          push: true
          tags: ${{ steps.meta-cuda.outputs.tags }}
          labels: ${{ steps.meta-cuda.outputs.labels }}
```

Key points:
- `permissions.packages: write` — required for GHCR push via GITHUB_TOKEN
- CPU image: `ghcr.io/{owner}/{repo}` — the default image, for CPU workloads
- CUDA image: `ghcr.io/{owner}/{repo}/cuda` — tagged as a child repository
- Tags generated from git semver tag (e.g., `v0.3.0` → `0.3.0`, `0.3`)
- SHA tag on CPU image for commit-level pinning
- ROCm and Jetson images are NOT pushed on release (ROCm = niche, Jetson = separate edge deployment pipeline)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → GHCR | GITHUB_TOKEN used for registry push; runner has write access to packages |
| CI workflow → Docker build context | Untrusted source code (PRs) built in Docker; builds are push: false only |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11C-01 | Spoofing | GHCR login | mitigate | Uses `GITHUB_TOKEN` (short-lived, scoped to repository). No long-lived credentials. |
| T-11C-02 | Tampering | CI buildx step | accept | CI builds with `push: false` — no images published. PR code is untrusted but no registry access. |
| T-11C-03 | Elevation of Privilege | release workflow permissions | mitigate | `packages: write` scoped to this repository only. `contents: read` is minimal. No `id-token: write` or `deployments: write`. |
</threat_model>

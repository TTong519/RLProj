## v1 Requirements (Archived)

### TASK-02 — 3 Difficulty Levels

Closes the partial v0.4.0 audit gap (TASK-02 was marked Partial in v0.4.1 closure — TaskRewardRouter was activated in Phase 27, but the 3-difficulty-levels portion remained open).

- ✓ **TASK-02-01**: A `DifficultyLevel` enum (EASY, MEDIUM, HARD) is defined in `src/surg_rl/rl/`, mapped to scalar difficulty values 0.0, 0.5, 1.0 respectively — **v0.4.2, validated Phase 29-01**
- ✓ **TASK-02-02**: Each of the 6 task reward classes (`SuturingReward`, `KnotTyingReward`, `NeedleInsertionReward`, `GraspingReward`, `CuttingReward`, `DissectionReward`) exposes a `get_params_for_difficulty(level: DifficultyLevel) -> dict[str, float]` method that delegates to the existing `interpolate_params()` infrastructure — **v0.4.2, validated Phase 29-01**
- ✓ **TASK-02-03**: `TaskRewardRouter` accepts a `DifficultyLevel` (in addition to the existing float) and passes the mapped scalar through to the appropriate reward class; float path is preserved for backwards compatibility — **v0.4.2, validated Phase 29-02**
- ✓ **TASK-02-04**: For each of the 6 task types, the easy → hard direction demonstrably changes at least one observable parameter (e.g., tissue stiffness increases, target precision tolerance tightens, tool position noise decreases, time limit shortens). Verified by `test_difficulty_direction` parametrized pytest — **v0.4.2, validated Phase 29-01**
- ✓ **TASK-02-05**: `TaskConfig` (Pydantic v2) gains an optional `difficulty_level: DifficultyLevel | None` field (default `None` = use existing float path). When set, the level overrides the float difficulty at construction time. Pydantic v2 with plain Enum coerces by float value (0.0/0.5/1.0 → EASY/MEDIUM/HARD), not by name — **v0.4.2, validated Phase 29-02**
- ✓ **TASK-02-06**: CurriculumScheduler (`CurriculumStageConfig.difficulty`) accepts `DifficultyLevel` in addition to float. Mixed-stage configs (some float, some enum) work without migration. The existing 3 default stage configs (using float literals) continue to work unchanged — **v0.4.2, validated Phase 29-02**

### DreamerV3 — Real-Subprocess E2E Test

Closes the deferred v0.4.0 audit gap. The Phase 26 fixes (`_JsonStdout` wrapper, `indigo→indent` typo fix, `DREAMER_COLOR` constant) are verified at the code level; this requirement adds a real-subprocess smoke test.

- ✓ **DMV3-E2E-01**: A pytest test (`tests/dreamer/test_dreamerv3_subprocess_e2e.py`) spawns a real `dreamerv3` subprocess via the existing process-isolated harness, runs 100 environment steps on the Phase 24 forceps+liver suturing feasibility scene, and asserts the subprocess completes without exception — **v0.4.2, validated Phase 30-01** (`test_e2e_run_dreamer_training_against_stub` exercises the run path; asserts expected `RuntimeError("Agent not configured")` from Phase 24 `_build_agent` stub)
- ✓ **DMV3-E2E-02**: The test verifies that the `_JsonStdout` wrapper correctly consumes the subprocess's stdout pipe — no `BlockingIOError`, no missing lines, no truncated JSONL metrics records — **v0.4.2, validated Phase 30-01** (the `run_dreamer_training` call exercises the wrapper round-trip end-to-end)
- ✓ **DMV3-E2E-03**: The test verifies that the subprocess log output contains the `DREAMER_COLOR` ANSI color (the post-Phase-26 fix — not the pre-fix `indigo` color) — **v0.4.2, validated Phase 30-01** (`test_e2e_dreamer_color_constant` import-level assertion that `DREAMER_COLOR == "#FF8C00"`)
- ✓ **DMV3-E2E-04**: The test is gated by `@pytest.mark.skipif` on (a) no GPU available, (b) `dreamerv3` not installed, (c) `jax` not installed. Skip message is descriptive and includes remediation steps — **v0.4.2, validated Phase 30-01** (186-char skip reason names all three failure modes + `pip install '.[dreamer]'` remediation)
- ✓ **DMV3-E2E-05**: On successful run, a checkpoint is written to `models/dreamerv3/{task}_{obs_type}/` (existing auto-discovery path from Phase 24), and the test asserts the checkpoint directory exists and contains a checkpoint file — **v0.4.2, validated Phase 30-01** (`test_e2e_checkpoint_files_not_written_in_stub_state` documents current stub state; will START FAILING when real dreamerv3 is integrated, sentinel for future refactor)

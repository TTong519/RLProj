## Overview

v0.4.2 is the second pure audit-gap-closure milestone, spawned by the v0.4.1 milestone closure. The v0.4.1 audit identified two residual items: TASK-02 (3-difficulty-levels) was left Partial after Phase 27 closed only the TaskRewardRouter activation portion, and the DreamerV3 real-subprocess E2E test was deferred because it requires GPU + `dreamerv3` install. v0.4.2 ships 2 surgical phases (3 plans total) that close both gaps without introducing new features.

The TASK-02 closeout adds a `DifficultyLevel` enum (EASY=0.0, MEDIUM=0.5, HARD=1.0) and threads it through the 6 task reward classes, `TaskRewardRouter`, `TaskConfig` (Pydantic v2), and `CurriculumStageConfig` ‚Äî all while preserving the existing float-based difficulty path. The DreamerV3 E2E closeout adds a single pytest test module gated by `(GPU + dreamerv3 + jax)` skipif, validating that the three Phase 26 fixes (`_JsonStdout` wrapper, `indent=2` typo, `DREAMER_COLOR = #FF8C00`) hold end-to-end when the real subprocess path runs.

**Audit verdict:** `passed` (11/11 v1 requirements satisfied, 0 partial, 0 deferred).

## Milestone Summary

**Key Decisions:**

- **D-29-01**: Single phase for TASK-02 is appropriate. The 6 requirements form one tightly-coupled feature: enum ‚Üí per-task methods ‚Üí router ‚Üí TaskConfig ‚Üí CurriculumScheduler. Subdividing into "implementation" and "integration" plans within Phase 29 is acceptable, but splitting into two phases would over-engineer a gap-closure milestone.
- **D-29-02**: `DifficultyLevel` is enum-only, not Pydantic-validated scalar. The enum is the public API; internally, only the scalar (`0.0`/`0.5`/`1.0`) is used. Avoids confusing double-validation (Pydantic validates enum membership; downstream uses scalar). Implemented as `_FloatMixin(float, Enum)` because Python's stdlib has no `FloatEnum` ‚Äî the mixin makes `DifficultyLevel.EASY == 0.0` True.
- **D-29-03**: No new `DifficultyLevelConfig` schema model. TaskConfig gets a single optional `difficulty_level` field; per-level override blocks (tissue_stiffness, target_precision_tolerance, tool_position_noise, time_limit) are deferred. The 6 `PARAM_BOUNDS` already encode the progression ‚Äî adding scene-level overrides is scope creep.
- **D-29-04**: `interpolate_params(difficulty)` remains the single source of truth. `get_params_for_difficulty(level)` is a thin wrapper that calls `interpolate_params(level.value)`. No duplicate math.
- **D-29-05**: Float path is preserved everywhere. `TaskRewardRouter(difficulty=0.5)` and `TaskRewardRouter(difficulty=DifficultyLevel.MEDIUM)` produce equivalent behavior. Mixed `CurriculumStageConfig.task_difficulty` (float in stage 1, enum in stage 2) works without migration.
- **D-29-06** (cycle resolution): Pydantic v2 + cross-package enum imports require a 3-step pattern: (1) `from __future__ import annotations` + string forward-ref annotation, (2) late import of the cross-package symbol at module bottom, (3) explicit `Model.model_rebuild()`. Additionally, breaking eager-load cycles uses lazy local imports inside the function body. This pattern is reusable for any future cross-package Pydantic v2 schema work and should be documented in CONVENTIONS.md.
- **D-29-07** (strict type check): The router normalization test uses `type(router._difficulty) is float` rather than `== 1.0` because the float-mixin on `DifficultyLevel` means `DifficultyLevel.HARD == 1.0` is True, which would mask the failure mode where the router stored the enum member instead of normalizing.
- **D-30-01**: Single-phase, single-plan E2E test. Splitting into "subprocess spawn" and "assertions" is over-engineering for a test-only phase; all 5 requirements describe the same test.
- **D-30-02**: macOS local run is expected to skip ‚Äî `STATE.md` Blocker #4 already documented this. The skip message must include `pip install '.[dreamer]'` as remediation so a developer with GPU can enable locally.
- **D-30-03**: Use the existing `DreamerSubprocess` and `_create_scene_for_task` infrastructure (Phase 24). Do not re-implement subprocess management in the test ‚Äî only orchestrate.
- **D-30-04**: Test file lives in `tests/dreamer/` (not `tests/`), consistent with the other DreamerV3 test files.
- **D-30-05**: Phase 30 verifies the Phase 26 fixes hold end-to-end ‚Äî the unit tests in `tests/test_dreamer_subprocess.py` (5 tests, `TestSubprocessStdoutProtocol`) and `tests/test_dreamer_training.py` (2 tests, `TestTrainingMetricsSave`) remain the primary regression coverage. The E2E test is a smoke test that runs the full code path on real hardware; it is the deferred DMV3-03 E2E validation.
- **D-30-06** (stub-state sentinel): The test asserts the EXPECTED `RuntimeError("Agent not configured")` from the current Phase 24 `_build_agent` stub, rather than asserting positive completion. This documents the current state and acts as a sentinel ‚Äî when real dreamerv3 is integrated, the test will START FAILING and must be flipped to assert positive completion.

**Issues Resolved:**

- TASK-02 partial close (v0.4.1) ‚Äî 3-difficulty-levels portion now fully closed: `DifficultyLevel` enum (EASY=0.0, MEDIUM=0.5, HARD=1.0) exists; all 6 task rewards expose `get_params_for_difficulty()` delegating to `interpolate_params()`; `TaskRewardRouter` accepts both `float` and `DifficultyLevel` (backwards compat preserved); `TaskConfig.difficulty_level` Pydantic v2 optional field added; `CurriculumStageConfig.difficulty` widened to `float | DifficultyLevel`; all 6 Phase 27 production scenes still load with `difficulty_level is None` (no migration).
- DMV3-03 deferred E2E (v0.4.1) ‚Äî `tests/dreamer/test_dreamerv3_subprocess_e2e.py` exists, gates on (GPU + `dreamerv3` + `jax`), exercises `_JsonStdout` wrapper round-trip + `DREAMER_COLOR` constant + checkpoint auto-discovery path; on macOS local it skips with a descriptive reason; on CI GPU host it runs and documents the Phase 24 stub state via the expected `RuntimeError`.
- Cycle resolution pattern (Phase 29-02) ‚Äî established a 3-step Pydantic v2 + cross-package enum import cycle pattern (string forward-ref + late import + `model_rebuild()`) plus a 4th step (lazy local imports inside function bodies) for cycles that the late import doesn't break. This pattern is reusable for any future cross-package Pydantic v2 schema work.
- Float-mixin `Enum` pattern (Phase 29-01) ‚Äî `_FloatMixin(float, Enum)` base class makes `DifficultyLevel.EASY == 0.0` True without requiring stdlib `FloatEnum` (which doesn't exist). Duck-typed `level.value` consumers mean the enum can be used transparently in tests and downstream code.

The scheduler accepts the enum but does not drive level progression (e.g., EASY ‚Üí MEDIUM ‚Üí HARD at stage boundaries). Future: feed enum values to `CurriculumStageConfig.difficulty` at stage transitions.

- **End-to-end `SurgicalEnv`-construction integration test for HARD fixture scene** ‚Äî Phase 29 code review finding WR-02. The fixture `suturing_difficulty_hard.json` is loaded and parsed, but no test actually constructs a `SurgicalEnv` from it end-to-end.
- **`CurriculumStageConfig.difficulty` normalization at env-construction** ‚Äî Phase 29 code review finding WR-03. The union `float | DifficultyLevel` is accepted but not normalized to `float` at the env boundary.

**Technical Debt Incurred:**

- **`CurriculumScheduler.current_difficulty` type lie** (Phase 29 code review WR-02 variant) ‚Äî The field is typed as `float` but can hold a `DifficultyLevel` enum when `CurriculumStageConfig.difficulty` is set to an enum. This is the price of union widening without normalization at the env boundary (deferred to v0.5.0+).
- **Test fixture scoping** (Phase 29) ‚Äî `suturing_difficulty_hard.json` is a copy of `scenes/simple_suturing.json` with one field added. A future deduplication step could use JSON pointer overlays instead of file copies, but that's a v0.5.0+ refactor.
- **Module-level pytestmark skipif** (Phase 30) ‚Äî The skipif evaluates at collection time, BEFORE the test bodies run. Heavy imports (e.g., `run_dreamer_training`) are inside test methods so a missing import inside the production module does not crash collection on macOS. This pattern works for current test code but is brittle if production imports move earlier.
